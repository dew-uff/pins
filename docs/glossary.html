Collection

This category classifies the provenance collection in 4 types: annotations, definition, deployment, and execution. These types are not mutually exclusive. For instance, many approaches use annotations as instrumentation to collect execution provenance. The following figure presents the collection taxonomy.

Annotations

Users can make annotations on procedures or on data. We classify annotations in 5 axis as following: placement, extraction, inclusiveness, target, and necessity.

Placement

This axis assumes two values: internal annotations and external annotations. Internal annotations occur inside the script definition. External annotations occur outside.

Extraction

This axis assumes two values: parseable annotations and executable annotations. Parseable annotations can be extracted statically, without the execution of the script. Executable annotations are executable code used to inform provenance.

Inclusiveness

This axis assumes two values: inclusive annotations and exclusive annotations. Inclusive annotations indicate the inclusion of data. Exclusive annotations filter data out of the collection.

Target

This axis assumes two values: definition and provenance. An annotation can describe the program definition by stating the relationships among structures and data. An annotation can also enrich the provenance itself by adding more data to it.

Necessity

This axis assumes two values: optional and mandatory. Mandatory annotations occur when the annotations are required to collect the provenance. Optional annotations occur when the approach can collect provenance without the annotations, but support annotations to enrich the collection.

Execution

Execution provenance corresponds to runtime logs and restrospective provenance. We identified 4 strategies of execution provenance collection: passive monitoring, overriding, post-mortem, and instrumentation.

Passive Monitoring

The passive monitoring strategy uses a tracer to observe the execution and log all low-level events during the execution. It imposes the biggest performance overhead, but it is also able to collect more provenance data.

Overriding

The overriding strategy automatically instruments the code to collect provenance. It finds patterns in the script (such as function calls), and replaces it to enriched versions that collect provenance.

Post-Mortem

The post-mortem strategy infers provenance from execution results after the execution. In order to collect this type of provenance, users need to specify the locations of output data and how it relates to input data. One way to apply the post-mortem strategy is to store all data files in a specific directory and collect all files before and after the execution.

Instrumentation

The instrumentation strategy requires users to change their code specifying what they want to collect.

Deployment

Deployment provenance represents the execution environment. It refers to operating system version, interpreter version, environment variables, dependencies to programs and modules, and all the remaining deployment information that describes the environment. Tools can collect this type of provenance before or during the execution.

Definition

Definition provenance refers to the project structure with scripts and input data. We classify definition provenance according to how and when it is collected.

How

Definition provenance can be collected by reading whole files or parsing files and extracting information from them.

When

Definition provenance can be collected statically, before or after the execution, or dynamically, during the execution.

Management

This category classifies provenance management features of the approaches in terms of storage, distribution, reproducibility, and versioning. The following figure presents the management taxonomy.

Storage

Provenance can be stored in database systems, transient memory, or files.

Database

As database systems, we found approaches that use Relational databases, Graph databases, and NoSQL databases.

Transient Memory

Approaches that use only the transient memory, consume the provenance during the script execution and do not store it.

File

As file storage, we found approaches that store provenance in interoperable files (e.g., PROV, OPM), log files, logic programming files (e.g., Prolog, Datalog facts), graph files (e.g., graphviz files), content databases (e.g., version control repositories), and proprietary file formats.

Distribution

After storing the provenance, the approaches can use files or repositories to distribute the provenance.

File

Approaches use the same file formats of file storage to distribute the provenance.

Repository

Repositories implement mechanisms for sharing the provenance and experiment definitions, encouraging the reuse of experiments of other people.

Reproducibility

This category indicates whether the approach supports reproducing an execution.

Versioning

Some approaches only collect the provenance of a single execution. Others consider that a script can evolve and collect the provenance of multiple executions. We identified three types of versioning support: Trial ID, Sequence, and Intention.

Trial ID

Approaches that support versioning by Trial ID only assign IDs to each execution, but provide no order of execution.

Sequence

Approaches that support versioning by Sequence describe the sequence of executions.

Intention

Approaches that support versioning by Intention support describing why and how the script and the provenance evolved.

Analysis

Provenance analysis aims at supporting the comprehension of data and processes. Analyzing provenance involves querying, visualizing, and comparing provenance data. The following figure presents the analysis taxonomy.

Query

We identified two types of queries: generic or specific.

Generic

This category indicates whether the approach supports a generic language for querying provenance, such as SQL, SPARQL, XQuery, Prolog, and Datalog.

Specific

This category indicates whether the approach supports a specific querying system for provenance, such as commands, functions, proprietary files, web interfaces, and interoperable formats (PROV/OPM) that can be queried in other provenance tools.

Visualization

We classify provenance visualization according to 3 axis: place, type, and completeness.

Place

The visualization can be internal or external. Internal visualization is implemented by the approach itself. External visualization occurs when the approach delegates the provenance visualization to an external tool, such as PROV visualizer.

Type

We identified four types of provenance visualizations: log, process, data, and combined.

Logs are usually a textual representation of the provenance data.

In a process graph, nodes represent processes/operations and arrows represent data.

In a data graph, nodes represent data and arrows represent processes/operations.

In a combined graph, nodes represent both processes/operations and data. Arrows represent their relationships.

Completeness

Some approaches present the complete provenance data. Others support summarizing it by clustering or filtering the data.

Comparison

Some provenance approaches support comparing data to present differences between results and for cache invalidation. Others support comparing provenance graphs to understand differences between executions.
